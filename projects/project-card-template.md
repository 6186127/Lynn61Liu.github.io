# Project Card Template

## 1. Project Header

### Project Name
A clear product or engineering-oriented title.

Examples:
- NZAT Workshop Management System
- WOF Inspection Mobile Workflow App
- AI-enabled Meal Planning Platform

---

### Project Type Badge

Use 1–2 standardised project type labels.

Examples:
- FULL-STACK PROJECT
- INTERNAL SYSTEM
- WORKFLOW PLATFORM
- AI-ASSISTED PROJECT
- CLOUD PROJECT
- MOBILE PROTOTYPE
- DISTRIBUTED SYSTEM

---

## 2. Project Summary

The summary should explain:

1. What the product/system is
2. What workflows or problems it handles
3. What operational/business value it creates
4. What engineering/system value it demonstrates

Recommended structure:

[System definition]

[Core workflows/features]

[Business or operational impact]

[Future scalability or engineering direction]

---

## 3. Engineering Highlights

Use 3–6 concise bullets describing technical and engineering value.

Examples:

- API-driven workflow architecture
- Dockerised deployment workflows
- Authentication and RBAC implementation
- Real-time workflow coordination
- Xero and email integration
- Testing and debugging practices
- Accessibility-aware UI implementation
- CI/CD deployment automation

Guidelines:
- Focus on engineering value
- Avoid generic feature descriptions
- Prefer architecture and workflow terminology

## 5. Reflection (Recommended)

Every project should include engineering reflection.

Recommended topics:
- Trade-offs
- Architecture lessons
- Workflow complexity
- Maintainability
- Scaling considerations
- Future improvements

Example:

Balancing operational flexibility with maintainable workflow structure required clearer separation between UI state and backend workflow state.

---

## 6. Future Iteration (Optional)

Explain what the next version would improve.

Examples:
- Role-based dashboards
- Better analytics visibility
- Event-driven workflow processing
- Mobile optimisation
- AI-assisted workflow automation
- Better testing coverage

---

## 7. Visual Direction

Project cards should:
- Be highly scannable
- Avoid long text blocks
- Use technology tags
- Use strong typography hierarchy
- Support responsive layouts
- Balance screenshots and engineering content

Avoid:
- Screenshot-only project cards
- Generic school-project descriptions
- Long paragraphs without hierarchy
- Technology dumping without business context

---

## 8. Core Principle

A project card should communicate:

1. What problem was solved
2. What engineering thinking was applied
3. What systems/workflows were designed
4. What technical depth exists
5. What lessons were learned
