# Portfolio Website AI Context

This document is the working context for AI tools editing or extending Lynn Liu's personal portfolio website.

Use it as the source of truth for positioning, tone, content boundaries, and product direction before changing `index.html`, `index.css`, case study content, project cards, or future article/project detail pages.

## 1. What This Website Is For

This portfolio is designed to support software engineering job applications in New Zealand.

Its job is not only to show a list of technologies, but to communicate:

- engineering thinking
- practical full-stack delivery
- strong frontend and UX awareness
- cloud-ready and backend-oriented growth
- maintainability, testing, and reliability mindset
- the ability to turn messy real-world workflows into usable digital systems

The ideal impression is:

> A thoughtful full-stack software engineer who can build practical systems with React, TypeScript, .NET, and modern engineering practices, while communicating clearly and learning fast.

## 2. Primary Audience

The main audience includes:

- hiring managers
- engineering managers
- technical recruiters
- senior developers reviewing junior-to-intermediate candidates
- companies hiring for practical product or internal-systems engineering roles

The website should especially support applications for:

- Full-Stack Developer
- Software Engineer
- Frontend Developer
- .NET Developer
- Cloud-ready Developer
- Internal Systems Developer
- Applied AI / AI-assisted Engineering roles

## 3. Core Positioning

Main positioning:

> Full-Stack Developer building cloud-ready, user-focused systems with React, TypeScript, .NET, and modern engineering practices.

Supporting strengths:

- strong frontend implementation and UX awareness
- practical backend and API development
- internal workflow and business application thinking
- growing cloud and DevOps capability
- AI-assisted engineering interest and experimentation
- strong troubleshooting mindset from enterprise support experience
- interest in testing, accessibility, maintainability, refactoring, and secure design

## 4. Tone And Brand

The site should feel:

- clean
- calm
- professional
- technically credible
- modern
- easy to scan
- slightly product-oriented, not just resume-like

The tone of writing should be:

- clear
- grounded
- reflective
- practical
- confident but not inflated

Avoid:

- exaggerated claims
- buzzword-heavy copy
- overly corporate language
- generic portfolio filler
- long dense text blocks
- playful effects that weaken credibility

## 5. Hard Content Rules

When editing this site, AI must follow these rules:

- Do not invent experience, metrics, companies, project scope, or certifications.
- Do not add claims of expertise stronger than the evidence on the site supports.
- Do not turn the portfolio into a generic "AI engineer" site unless the content is backed by real projects or learning.
- Keep the strongest emphasis on full-stack engineering, product-minded implementation, and practical systems work.
- Preserve Lynn's existing mix of engineering, UX awareness, and reflective writing.
- If a section is aspirational or planned, label it clearly instead of presenting it as already built.

## 6. Current Homepage Structure

The current homepage sections in `index.html` are:

1. Hero
2. Skills
3. Experience
4. Featured Projects
5. Engineering Articles
6. Contact

This structure is already strong and should generally be preserved unless there is a clear reason to change it.

## 7. Section Intent

### Hero

Purpose:

- quickly communicate who Lynn is
- show full-stack positioning
- highlight backend/cloud-ready direction
- show current learning focus

Current message direction:

- full-stack development
- React + TypeScript
- .NET backend
- cloud-ready systems
- practical workflow and product thinking

The hero should feel concise, credible, and strong.

### Skills

Purpose:

- show grouped engineering capabilities instead of a shallow keyword dump
- frame skills as ways of building and shipping software
- balance implementation skills with systems thinking

Skills should usually be grouped by capability areas, not just by tool names.

### Experience

Purpose:

- present work history in an engineering-oriented way
- emphasise contribution, workflow impact, collaboration, and delivery

Experience bullets should favour:

- built
- improved
- designed
- supported
- integrated
- documented
- collaborated

Avoid resume bullets that only list responsibilities.

### Featured Projects

This is one of the most important sections.

Projects should communicate:

- the problem
- the system or product idea
- the implementation approach
- the stack used
- a meaningful challenge
- what was learned

Projects should be presented as engineering case studies, not just visual cards.

### Reflection Articles

Purpose:

- show reflection
- show system thinking
- show trade-off awareness
- strengthen engineering credibility

This section should stay curated. Quality matters more than quantity.

### Contact

Purpose:

- clearly show Lynn is open to software engineering opportunities in New Zealand
- make contact easy
- end the site with warmth and professionalism

## 10. Visual Direction

The visual direction should remain:

- minimal but not bland
- spacious
- readable
- card-based where useful
- lightly polished rather than flashy

The current direction is close to:

- soft neutral backgrounds
- clear typography
- restrained accent colour
- structured sections
- minimal motion

Avoid:

- heavy gradients or loud colour palettes
- startup-style hype visuals
- excessive animation
- cramped layouts
- decorative UI that distracts from content

## 11. Writing Guidance For AI

When writing or rewriting portfolio copy:

- prefer plain, strong sentences
- make each paragraph earn its place
- connect technologies to outcomes
- frame growth areas honestly
- sound like an engineer explaining real work, not a marketer

Good examples of emphasis:

- workflow clarity
- maintainability
- API-driven design
- delivery confidence
- debugging and troubleshooting
- accessibility and usability
- operational reliability

Less helpful emphasis:

- listing too many tools without context
- calling everything "innovative"
- generic claims like "passionate developer" without proof

## 12. Editing Guidance For Future AI Sessions

Before making changes:

1. Read this file first.
2. Check whether the requested change is about current reality or future direction.
3. Verify the relevant section in `index.html` and `index.css`.
4. Preserve factual accuracy.

When adding new content:

- keep structure scannable
- prefer meaningful section headings
- use specific technology names only when they add value
- keep project and experience descriptions tied to business or user outcomes

When uncertain:

- choose the more grounded version
- avoid overstating seniority
- preserve credibility over impressiveness

## 13. Short Summary For AI

If you only remember a few things, remember this:

- This is a professional software engineering portfolio for New Zealand job applications.
- The strongest positioning is practical full-stack engineering with React, TypeScript, .NET, cloud-ready thinking, and user-focused implementation.
- The site should feel credible, clean, reflective, and product-aware.
- Do not invent achievements or over-market AI capability.
- Projects, experience, and engineering thinking matter more than decorative design.
